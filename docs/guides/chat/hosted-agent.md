# 托管智能体（Hosted Agent）

托管智能体是 EvoFlow 的**长程自动化能力**——你下达一个目标，主智能体自行规划、执行、自检、调整，按步数或时长边界自动停止；中途可暂停、补充指令、终止，结束后可推送结果到飞书等渠道。适合"一次交代、长时间无人值守完成"的任务。

---

## 一、何时使用托管

| 场景 | 适合托管 |
|------|---------|
| 定期巡检 / 监控 / 数据汇总 | ✅ 设小时级或天级步数 |
| 长任务的迭代式推进（写论文、做调研、改造工程） | ✅ 让它自驱多轮推进 |
| 一次性问答 / 简短代码改造 | ❌ 普通对话即可 |
| 严格按时间触发的周期任务 | ❌ 用 [自动化](../tasks/scheduled-tasks.md) 更合适 |
| 多角色协作的工程交付 | ❌ 用 [任务中心](../tasks/task-center.md) + 项目团队 |

**托管 vs 自动化**：托管是"持续推进一件事直到完成或边界"，定时是"到点跑一次"。

**托管 vs Plan 模式**：Plan 是"单会话内拆步执行"，托管是"自驱多轮直到边界"。

## 二、入口与首页

托管的两个入口都在聊天页：

| 入口 | 位置 |
|------|------|
| **托管按钮** | Composer（底部输入框）右上角的「托管」按钮 |
| **托管面板** | 启动后聊天页右侧出现的常驻面板，实时显示状态、轮次、最近一轮摘要 |

**托管面板布局**：

- **顶部状态徽标**：`idle / running / waiting / paused / error` 五种状态
- **进度指示**：当前轮次 / 最大轮次 + 已运行时长
- **快捷操作**：暂停 / 恢复 / 终止
- **运行日志**：每轮工具调用与摘要，可展开查看完整调用链
- **产物列表**：本次托管产生的 `outputs/` 文件，可点开预览

## 三、创建托管任务

### 方式一：界面创建（推荐）

1. 在任意聊天页 Composer 右上角点「**托管**」按钮，右侧弹出配置面板
2. 填写配置项（见 [四、配置字段](#四配置字段)）
3. 点击「**启动托管**」，后台立即开始第一轮

### 方式二：对话触发

不想离开聊天主流时，直接描述需求：

```
帮我创建一个托管任务，每小时检查一次服务器健康，运行 7 天后自动停止，
完成时推送总结到飞书运维群。
```

主智能体识别后会返回一个**托管方案确认条**，含三个选项：

| 选项 | 行为 |
|------|------|
| **填入并开始** | 直接落到托管面板并启动 |
| **仅填入** | 只把方案塞进面板，稍后手动启动（适合先调一调再跑） |
| **取消** | 不创建 |

> ⚠️ 主智能体**不会绕过确认条自行启动托管**——这是用户授权机制。

## 四、配置字段

| 字段 | 必填 | 默认 | 范围 | 说明 |
|------|------|------|------|------|
| **任务名称** | 否 | 空 | — | 用于在托管历史里识别 |
| **任务目标 prompt** | 是 | — | 1–4000 字 | 自然语言描述要达成的最终状态，**越具体越好** |
| **最大轮次 max_steps** | 是 | `8` | 1–1000 | 调度模型最多执行多少轮；与时长边界**先到先停** |
| **自动停止 auto_stop_minutes** | 否 | `0` | 0–10080 | 单位分钟；`0` = 不启用该计时，最长 7×24h |
| **每轮间隔 step_delay_ms** | 否 | `1200` | ≥200 | 两轮之间的间隔毫秒，避免太频繁 |
| **重试上限 retry_limit** | 否 | `2` | ≥0 | 连续失败后的最大重试次数 |
| **主动性 initiative** | 否 | `70` | 0–100 | 主动决策的倾向：越高越敢自己拍板，越低越频繁问用户 |
| **情绪智能 emotional_intelligence** | 否 | `true` | 布尔 | 是否启用情绪感知（调整回复语气以匹配用户情绪） |
| **人格风格 persona_style** | 否 | `professional` | `warm / professional / lively / calm` | 影响调度提示词与推送文案口吻 |
| **结束推送 feishu_push_on_complete** | 否 | `true` | 布尔 | 自然结束时向飞书默认会话推送 Markdown 总结 |
| **持续进化** | 否 | 关 | 布尔 | 启用 `self-evolution` 思路：长任务里允许自改提示词、改技能（**高级选项**，仅受控实验场景启用） |

> **关于 `max_steps` 与 `auto_stop_minutes`**：两个边界**任一达到就停**。常见组合：
>
> - 短任务：`max_steps=8`, `auto_stop_minutes=0` — 跑完 8 轮就停（默认）
> - 长任务定时长：`max_steps=1000`, `auto_stop_minutes=4320`（3 天）— 谁先到先停

## 五、运行特性

- **最长 7×24h** 不间断运行，达到时长边界自动停止
- 主控需要补充信息时**自动转为 `waiting` 状态**，最长等待 7 天；用户答复后回到 `running`
- **双停止规则**：`max_steps` / `auto_stop_minutes`，任一满足就停
- 长任务上下文**自动压缩**：最近 10 轮完整保留，更早的转为摘要，避免 Token 溢出
- 中途产生的产物（文件、报告、截图）落到 `outputs/`，可在对话里用 `@@outputs/xxx@@` 引用
- 状态机 5 态：

```
idle ──启动──▶ running ──需补充信息──▶ waiting ──用户答复──▶ running
                  │                              │
                  ├──手动暂停──▶ paused ──恢复──▶ running
                  │
                  ├──达到边界──▶ idle（自然结束）
                  │
                  └──严重错误──▶ error
```

## 六、使用示例

### 示例 1：服务器健康巡检

```
帮我创建一个托管任务，每小时检查一次以下服务的健康状态：
- 网关：http://localhost:8001/api/health
- LangGraph：http://localhost:8000/health
- Postgres：psql -c "SELECT 1"

发现异常时立即整理告警信息推送到飞书运维群（chat_id: oc_xxx），
连续 24 小时未发现异常时每天 9:00 推送一次"全部正常"汇总。
运行 7 天后自动停止。
```

**适合字段**：`max_steps=200`, `auto_stop_minutes=10080`, `initiative=80`, `persona_style=calm`

### 示例 2：竞品调研报告

```
帮我做一份关于"国产 AI Agent 平台"的竞品调研报告，覆盖产品定位、
能力矩阵、定价、最近 3 个月动态。每次推进时把新增信息整合进 outputs/research.md，
推进到自认为充分（覆盖至少 8 家厂商）时停止。
```

**适合字段**：`max_steps=30`, `auto_stop_minutes=0`, `initiative=60`, `persona_style=professional`

### 示例 3：长任务工程改造

```
帮我把 backend/app/channels/ 下飞书渠道的消息格式化逻辑重构成独立模块，
要求：
1. 拆出 feishu_message_format/ 子目录，按消息类型分文件
2. 补类型注解
3. 加 pytest 单测覆盖核心格式化函数
4. 跑测试通过后才算完成

完成时推送总结到飞书并附带 outputs/refactor-summary.md
```

**适合字段**：`max_steps=50`, `auto_stop_minutes=720`, `initiative=70`, `persona_style=professional`

## 七、运行时管理

| 操作 | 效果 |
|------|------|
| **暂停** | 进入 `paused` 状态，保留上下文，下次接着干 |
| **恢复** | 从 `paused` 回到 `running`，从上一轮的安全断点接续 |
| **终止** | 立即停止并标记 `idle`，已产生的产物全部保留 |
| **补充指令** | 在聊天框追加消息，主控下一轮自动并入目标 |
| **改配置** | 暂停状态下可在面板里改 `max_steps`、`auto_stop_minutes`、`persona_style`，恢复后生效 |
| **查看实时日志** | 展开"运行日志"，看每轮工具调用与摘要 |

终止或自然结束后，**全部执行记录会沉淀到当前会话的消息流**，可滚动回看每一轮决策。

## 八、飞书 / IM 协同

如果配置了 `feishu_push_on_complete=true` 且飞书渠道已就绪：

- 托管自然结束 / 被终止 / 异常停止 → **自动推送 Markdown 总结**到默认会话（与自动化同源：config 配置或入站学习的 chat_id）
- 飞书群内发送 `开始` / `确认` / `继续` 等指令，可在 EvoPanel 在线且已连接网关时**触发桌面侧应用托管方案并启动**（联动详情见 [飞书集成](../integration/feishu-integration.md)）

**推送内容包含**：

- 任务目标摘要
- 总耗时、总轮次、停止原因
- 关键产出物（含 `outputs/` 路径）
- 错误次数与最后一次错误（如有）

## 九、与自动化、Plan、任务中心的组合

四种任务能力的核心差异：

| 维度 | 托管 | 自动化 | Plan 模式 | 任务中心 |
|------|------|---------|----------|---------|
| **触发** | 立即开始，连续多轮 | 到点触发，单次 | 用户确认后开始 | 创建后队列接手 |
| **持续时间** | 长（最长 7×24h） | 单次 | 单会话内 | 跨多个会话 |
| **目标特征** | 一个开放目标 | 一段确定脚本 | 多步骤工程 | 多角色协作工程 |
| **协作粒度** | 单 Agent 自驱 | 单 Agent 单次 | 单 Agent 多步 | 多 Agent DAG |

**最重组合**：**自动化**到点启动一次**托管**，托管内部又走 **Plan 模式**派发子任务，对应每天/每周自动产出复杂工程性产物的场景（如自动周报、自动竞品监控报告）。

## 十、常见问题

**Q：托管不动了，没报错也没下一轮？**
打开"运行日志"看最近一轮：常见是模型连接失败 → 检查模型配置；或主控转 `waiting` 等待人工答复 → 在聊天框补一句即可继续；或达到 `auto_stop_minutes` 自然结束。

**Q：能不能让多个托管并行？**
当前**一个会话同时只允许 1 个活跃托管**；要并行多个任务请开多个会话或拆 Plan 子任务。多会话并行运行时模型并发受 Gateway 配置限制（详见 [自动化任务调度](../tasks/automation-scheduler.md)）。

**Q：托管会改自己的提示词 / 加技能吗？**
普通模式**不会**——主智能体只在用户授权的工具集内工作。启用「持续进化」后才会允许 `self-evolution` 类操作，建议仅在受控实验场景启用。

**Q：托管期间能切换主智能体角色吗？**
**不能**。托管启动时绑定的就是当前角色与场景，整个生命周期不变。想换角色请先终止当前托管，切换后再发起新的。

**Q：报"主控连接失败 / model error"怎么办？**
托管会按 `retry_limit` 自动重试，连续失败超过阈值后转入 `error` 状态。常见原因：

- 模型 API key 失效或额度耗尽 → 「设置 → 模型」检查
- 网络抖动 → 等几分钟手动恢复
- 模型不支持工具调用 → 切换到支持 function calling 的模型

**Q：终止后能不能续跑？**
能。终止后会话消息流里保留完整上下文，重新启动一个托管并把目标里加一句"基于刚才的产物继续"即可。

**Q：模型成本怎么控？**
三个手段：

1. 调小 `max_steps`（默认 8 已经偏小，长任务才需要调大）
2. 调大 `step_delay_ms`（巡检类任务设 60000ms 起，每分钟一轮）
3. 给主控选成本更低的模型（如 DeepSeek V3 / 通义 Plus 等）
4. 关掉非必要的工具白名单（特别是 `web_search` 与生图类工具）

**Q：推送内容能改吗？**
当前推送模板由 Gateway 渲染，不开放定制。如果需要自定义格式，可在任务目标里要求"完成时把总结写到 `outputs/summary.md`"，然后推送内容会附带该文件链接。
