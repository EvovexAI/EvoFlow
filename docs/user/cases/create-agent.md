# 案例二：雇佣代码审查员——自动 PR 审查

> **场景：团队每天有多个 PR 提交，人工审查费时又容易遗漏。**
>
> 雇一个"代码审查员"员工，自动检查新提交的代码质量，输出审查意见。常规问题自动处理，重大设计问题才问你。

## 操作步骤

### 1. 创建代码审查角色

在「智能体」页创建角色，关键配置：

**SOUL 示例：**

```
你是一个资深的代码审查工程师，精通多种编程语言和架构模式。
审查时遵循：先理解变更意图，再逐行检查代码质量，
最后从可维护性、性能、安全性三个维度给出评价。
代码风格问题给出修改建议，架构问题标记为"需讨论"。
```

**工具白名单：** 只勾选 `read`、`rg`、`search_code_index`、`git`（只读操作）——**不给** `write` / `delete` / `terminal`（写权限），确保审查员只能看不能改。

### 2. 雇佣为员工

| 字段 | 填写示例 |
|------|----------|
| **岗位名称** | 代码审查 |
| **职责** | 每 2 小时检查项目仓库新提交，执行代码审查，输出审查意见；遇到重大设计问题标记为"需讨论" |
| **管辖工作空间** | 项目仓库目录 |
| **自主权** | **平衡型**（常规审查自动完成，设计问题才问你） |
| **上岗节奏** | 每 2 小时 |

### 3. 审查结果

员工自动执行后，审查意见会写到工作空间。你可以在「工作项」看板查看：

```
审查报告：2025-07-22 10:00
文件：src/api/handler.ts
问题：
  - 第 42 行：未处理空指针异常（严重）
  - 第 78 行：硬编码配置值，建议抽到环境变量（建议）
  - 第 120 行：函数超过 80 行，建议拆分（风格）
```

### 4. 调优技巧

| 需求 | 做法 |
|------|------|
| 让审查更严格 | 在 SOUL 里加"安全漏洞必须标记" |
| 只看特定目录 | 在岗位配置里设「关注子路径」为 `src/` |
| 审查结果推送飞书 | 在职责里写"审查完成后推送到飞书群" |
| 不想让员工自动改代码 | 工具白名单里**不勾** `write` / `replace` / `delete` |

## 要点

- 审查员只读不写：工具白名单是安全边界，不是摆设
- 平衡型自主权 = 常规审查自动跑，只有"需讨论"级别的问题才等你批
- 如果员工是审查项目新提交，确保工作空间绑定了正确的项目目录
