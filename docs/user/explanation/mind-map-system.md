# 思维导图系统

> **思维导图就是 AI 的"思考过程可视化"**。
>
> **比如 AI 在跑"调研 3 家 AI Agent 平台"这个任务时**，你打开右侧思维导图面板会看到：
> - 总目标：调研 3 家 AI Agent 平台
> - 分支一：LangGraph ✅ 已查完（结论：支持 DAG 编排，Python 原生）
> - 分支二：CrewAI 🔄 正在查（状态：搜索中）
> - 分支三：AutoGen ⏳ 待查
> - 卡点：某个分支显示"找不到定价信息"
>
> 聊天记录看"说了什么"，思维导图看"在想什么、追哪条线"。适合排查长任务卡顿、回看决策链。
>
> **功能关系**：思维导图与[记忆系统](memory-system.md) 互补——记忆系统解决跨会话的用户偏好记忆，思维导图解决单会话的调查链路可追溯性。思维导图由 AI 在排查、读代码、改文件的过程中自动构建，是 [Agent 系统](agent-system.md) 工作过程中的辅助记录层，帮助 AI 在长会话中不丢失上下文和证据链。

## 背景

在与 AI 助手的长会话协作中，经常会遇到几个痛点：

- **对话越来越长，关键信息被淹没**——查了哪些文件、发现了什么证据、验证了哪些假设，散落在海量对话轮次中，难以回溯。
- **上下文压缩后"失忆"**——长会话会被系统压缩以控制 token 消耗，压缩后对话细节丢失，AI 可能忘记之前的发现。
- **多线索调查混乱**——复杂问题往往需要同时追多条线（前端链路、后端逻辑、数据库层），结论混在一起分不清哪条线查到哪了。
- **结论缺乏证据支撑**——AI 说"已修复"，但你不知道它是怎么得出这个结论的，无法验证是否真的修好了。

思维导图系统就是为解决这些问题而设计的：**在会话过程中自动构建一棵可追溯的知识与逻辑图谱**，把调查链路的每一步——文件、函数、假设、证据、结论——都结构化地记录下来。

## 设计理念

思维导图系统的核心设计原则是**实时自动构建、证据驱动、分支独立管理**：

1. **实时自动构建**：不需要用户手动创建或维护，AI 在排查、读代码、改文件的过程中自动更新图谱。
2. **证据驱动**：每一个结论节点都必须有对应的文件/函数/假设节点作为证据支撑——没有证据就不能下结论。
3. **分支独立管理**：每条调查线是一个独立的分支，可以单独打开、关闭、标记为已解决或已排除，互不干扰。

## 它解决了什么问题

### 1. 长会话"失忆"问题

长会话会被系统压缩（context compaction），聊天记录的细节会丢失。思维导图是压缩之外的第二份记忆——文件路径、函数签名、行号、根因结论都以结构化数据保留着，不会因为对话变长而遗忘。

**实际场景**：一个排查持续了 50 轮对话，中间经历了上下文压缩。如果没有思维导图，AI 可能会忘记第 5 轮发现的某个关键证据；有了思维导图，那个证据作为 `file:` 节点始终存在，AI 每一轮都能看到。

### 2. 调查链可追溯

普通 AI 助手说"问题在 `auth.ts` 里"，你不知道它怎么得出这个结论的。思维导图里你能看到完整链条：

```
flow:auth-error → file:src/auth.ts (verifyToken L42 缺少 Bearer 前缀)
                → gap:jwt-decode (假设：token 解码失败) → refuted（已排除）
                → claim:auth-fix (根因：缺类型守卫，已修复验证)
```

每一步都有证据支撑，不是空口结论。你可以从任何一个结论节点反向追溯到支撑它的文件和验证过程。

### 3. 多分支不混乱

复杂问题往往要同时追多条线。比如一次排查可能同时追了「前端发送链路」「后端模型创建」「factory 映射」三条线。每条线是一个独立的 `flow` 节点，可以：

- **单独关闭**：一条线查完了，标记为 `resolved`，不再出现在进行中列表。
- **独立标记**：验证通过标 `verified`，被排除标 `refuted`，外部阻塞标 `blocked`。
- **不会混线**：多条线的结论各自挂在自己的分支下，不会混在一起。

### 4. 结论有据可查

AI 告诉你"已修复"之前，思维导图里必须有对应的 `claim:` 结论节点 + 支撑它的 `file:`/`gap:` 证据节点。这从机制上防止了"嘴上说修好了，实际没验证"的问题。

**实际场景**：AI 说"登录 401 已修复"，但思维导图里 `flow:fix-401` 还是 `active` 状态、没有 `claim:` 节点——这就说明结论还没经过验证，你一眼就能看出来。

### 5. 用图表说话

对于调用链、时序交互、状态机这类逻辑，纯文字说不清楚。思维导图支持嵌入 Mermaid 图表节点，在面板里直接渲染：

| 场景 | 图表类型 | 示例 |
|------|----------|------|
| 函数调用链 | flowchart 流程图 | `chatSend → buildRunContext → hardGuard → sendToBackend` |
| API 交互过程 | sequence 时序图 | 前端 → Gateway → LangGraph → LLM 的请求时序 |
| 状态流转 | state 状态图 | 会话模式切换：auto ↔ manual ↔ flash |
| 模块关系 | class 类图 | Middleware 各子类的继承与组合关系 |
| 数据表关系 | ER 图 | sessions / threads / messages 的外键关系 |
| 部署架构 | architecture | Nginx → Gateway → LangGraph Server → Sandbox |

## 核心概念

### 节点类型

思维导图由不同类型的节点组成，每种节点承载不同的信息：

| 节点类型 | 前缀 | 用途 | 示例 |
|----------|------|------|------|
| **目标** | `goal:` | 会话的总体目标，每个会话只有一个 | "修复登录页 401 错误" |
| **分支** | `flow:` | 一条独立的调查线 / 子问题 | "排查前端发送链路" |
| **文件** | `file:` | 查阅或修改过的代码文件，记录路径 + 关键函数 + 行号 | `src/auth.ts — verifyToken() L42 缺少 Bearer 前缀` |
| **函数** | `fn:` | 具体函数的签名与关键逻辑 | `def create_chat_model(name, thinking_enabled)` |
| **数据** | `data:` | 数据结构、配置项、消息格式 | `ToolMessage.content (str)` |
| **假设** | `gap:` | 待验证的猜想，有状态标记 | "假设：JWT 解码失败导致 401" → `refuted` |
| **结论** | `claim:` | 经过验证的最终结论 | "根因：verifyToken 缺少 Bearer 前缀检查" |
| **图表** | `diagram:` | Mermaid 源码，面板内直接渲染 | `flowchart TD; A --> B` |
| **笔记** | `note:` | 临时记录的补充信息 | "需确认：gateway 层是否透传 thinking_type" |

### 边关系

节点之间通过边连接，表示它们之间的关系：

| 关系 | 含义 | 典型用法 |
|------|------|----------|
| `contains` | 包含 | flow → file（分支包含文件） |
| `calls` | 调用 | fn → fn（函数 A 调用函数 B） |
| `imports` | 导入 | file → file（文件 A 导入文件 B） |
| `reads` / `writes` | 读 / 写 | file → data（文件读取某数据） |
| `depends` | 依赖 | flow → gap（分支依赖某假设验证） |
| `defines` | 定义 | file → fn（文件定义了某函数） |
| `part_of` | 属于 | flow → flow（子分支属于父分支） |

### 节点状态

分支（`flow:`）和假设（`gap:`）节点有生命周期状态：

| 状态 | 含义 | 说明 |
|------|------|------|
| `active` | 进行中 | 默认状态，正在调查 |
| `resolved` | 已解决 | 问题已查明 / 已修复，分支关闭 |
| `verified` | 已验证 | 结论经过实际验证（测试通过 / curl 确认） |
| `refuted` | 已排除 | 假设被推翻，这条线不用再追 |
| `blocked` | 被阻塞 | 遇到外部阻碍（缺权限 / 依赖未就绪） |
| `parked` | 已搁置 | 暂不处理但不声称已解决，用户可随时恢复 |
| `collapsed` | 已折叠 | 已确认解决但暂时归档，需要时可展开回看 |

## 图谱结构

一个完整的会话思维导图是一棵**树 + 交叉连线**的结构：

```
🎯 goal:session —— 会话目标
  └─ 🔄 flow:auth-error —— 排查认证错误
       ├─ 📄 file:src/auth.ts —— verifyToken() L42
       ├─ 📄 file:src/jwt.ts —— decodeToken() 
       ├─ ⚡ fn:verify_token —— def verify_token(token: str) -> dict
       ├─ ❓ gap:jwt-decode —— "假设：token 解码失败" → refuted
       ├─ ✅ claim:auth-fix —— "根因：缺 Bearer 前缀检查"
       └─ 📊 diagram:auth-flow —— flowchart 调用链
  └─ 🔄 flow:db-connection —— 排查数据库连接（另一条独立线）
       ├─ 📄 file:db/pool.ts
       └─ ✅ claim:db-fix —— "连接池未释放"
```

树状结构保证层级清晰，交叉连线（如 `file:auth.ts → imports → file:jwt.ts`）保证真实依赖关系不丢失。

## 工作流程

思维导图在会话中的构建分为三个阶段：

### 阶段一：建立目标与分支（探索期）

会话开始时，AI 首先设定 `goal:` 节点明确本次会话目标，然后为每个子问题创建 `flow:` 分支。此时节点可能是空壳（占位），还没有具体内容。

### 阶段二：实时追加证据（调查期）

AI 在排查过程中——读文件、搜索代码、跑命令——每一步都会把发现追加到对应的节点：

- 读到一个文件 → 创建 `file:` 节点，记录路径、关键函数、行号
- 发现一个可疑点 → 创建 `gap:` 假设节点
- 验证了一个假设 → `patch_node` 更新 `gap:` 状态为 `resolved` 或 `refuted`
- 调用了某个函数 → 创建 `fn:` 节点，记录签名和关键逻辑
- 需要画图说明 → 创建 `diagram:` 节点，写入 Mermaid 源码

### 阶段三：关闭分支并出结论（收尾期）

当一条调查线完成时：

1. 将 `flow:` 节点状态改为 `resolved` / `verified` / `blocked`
2. 在该 flow 下创建 `claim:` 结论节点，写明最终结论
3. 如果分支下有 `gap:` 节点，逐一更新状态为 `resolved` 或 `refuted`

**关闭分支时，AI 用覆盖写入（body）写最终结论，而不是追加——避免旧描述与新结论矛盾共存。**

## 与普通笔记的区别

| 对比维度 | 普通笔记 / 聊天记录 | 思维导图 |
|----------|---------------------|----------|
| 结构 | 平铺列表，线性排列 | 树 + 图，有父子层级有交叉连线 |
| 实时性 | 手动记录或散落在对话中 | AI 边干活边自动更新 |
| 可追溯 | 只有结论，过程丢失 | 结论 + 证据 + 验证状态，完整链条 |
| 分支管理 | 全混在一起 | 每条线独立开/关，状态分明 |
| 图表 | 无 | 内嵌 Mermaid 可视化 |
| 上下文压缩后 | 细节丢失 | 结构化数据独立保留，不受压缩影响 |
| 防止虚假结论 | 无机制约束 | 必须有证据节点才能下结论 |

## 与记忆系统的关系

思维导图和[记忆系统](memory-system.md)都涉及"会话信息的持久化"，但定位不同：

| | 记忆系统 | 思维导图 |
|---|---|---|
| **时间维度** | 跨会话——从多次对话中提取长期事实 | 单会话——记录本次调查的完整链路 |
| **信息类型** | 用户偏好、工作背景、个人事实 | 文件路径、函数签名、假设、证据、结论 |
| **构建方式** | 后台异步提取，LLM 总结 | 实时同步构建，AI 工作时即写 |
| **用途** | 让 AI 记住"你是谁、你喜欢什么" | 让 AI 记住"这次排查查了什么、发现了什么" |
| **生命周期** | 长期保留，跨会话复用 | 会话内有效，会话结束后可归档 |

两者互补：记忆系统解决"跨会话连续性"，思维导图解决"单会话可追溯性"。

## 使用方式

思维导图由 AI 自动构建，用户通常不需要手动操作。但在 EvoFlow 面板中，你可以：

- **查看图谱**：在会话面板中查看实时构建的思维导图树形结构
- **查看图表**：点击 `diagram:` 节点查看渲染后的 Mermaid 图表
- **恢复搁置分支**：右键 `parked` 状态的节点，改回 `active` 重新调查
- **展开折叠分支**：将 `collapsed` 状态的节点改回 `active` 展开回看

## 最佳实践

1. **复杂排查交给 AI**：涉及多文件、多链路的问题最适合用思维导图——AI 会自动拆分分支、逐条追踪。
2. **关注分支状态**：如果 AI 说"已解决"但对应 `flow:` 还是 `active`，说明可能还没真正验证完。
3. **善用图表节点**：遇到调用链复杂的情况，可以让 AI 生成 `diagram:` 节点可视化。
4. **不要手动维护**：思维导图是 AI 自动构建的，手动编辑容易破坏图的一致性约束。
