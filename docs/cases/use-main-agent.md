# 案例一：在对话中使用主智能体

## 场景说明

你已经安装并配置好了 EvoFlow，想要和默认的主智能体（Lead Agent）进行对话，让它帮你解答问题、分析内容或执行任务。

## 前置准备

- 已完成 [安装](../getting-started/installation.md)
- 已配置至少一个 AI 模型（API Key）
- 已启动 EvoPanel 并登录

## 操作步骤

### 步骤 1：打开对话页面

在 EvoPanel 左侧导航栏中，点击 **「对话」** 图标，进入对话页面。

<!-- 截图占位：use-main-agent-step1.png — 打开对话页面 -->

> **截图说明**：展示 EvoPanel 左侧导航栏，标注"对话"图标的位置。

### 步骤 2：选择对话模式

在对话页面上方，你会看到四种模式选择：

| 模式 | 适用场景 |
|------|----------|
| **聊天** | 快速问答、简单咨询 |
| **规划** | 需要深度分析、多步骤思考的问题 |
| **执行** | 需要调用工具或委托子智能体完成的任务 |
| **无限** | 全部能力开放，适合复杂场景 |

选择一个适合你当前需求的模式。

<!-- 截图占位：use-main-agent-step2.png — 选择对话模式 -->

> **截图说明**：展示对话模式选择区域，标注四种模式的切换按钮。

### 步骤 3：输入你的问题

在底部输入框中输入你的问题或指令，按回车发送。

**示例提示词：**

```
帮我分析一下目前 Python 项目中依赖管理的最佳实践，
有哪些主流方案？各自的优缺点是什么？
```

<!-- 截图占位：use-main-agent-step3.png — 输入问题 -->

> **截图说明**：展示输入框输入问题的界面，标注发送按钮。

### 步骤 4：查看流式响应

主智能体会以流式方式输出回答，支持 Markdown 渲染。如果智能体调用了工具（如搜索、文件读取），你可以在对话中看到工具调用的过程。

<!-- 截图占位：use-main-agent-step4.png — 查看流式响应 -->

> **截图说明**：展示智能体流式输出回答的界面，标注工具调用可视化区域。

### 步骤 5：多轮对话

你可以继续追问、要求补充或让智能体调整回答方向。主智能体会保持上下文连贯性。

**示例提示词：**

```
请帮我用表格的形式对比这几种方案，
并推荐一个最适合中型团队的选择。
```

<!-- 截图占位：use-main-agent-step5.png — 多轮对话 -->

> **截图说明**：展示多轮对话的完整界面，标注上下文保持的效果。

### 步骤 6：上传文件或多模态输入（可选）

你可以粘贴截图或拖拽图片到对话中，主智能体会进行图片识别和分析。

<!-- 截图占位：use-main-agent-step6.png — 上传图片 -->

> **截图说明**：展示上传图片后智能体识别和分析的界面。

## 小结

完成以上步骤后，你已经掌握了与主智能体对话的基本方式。主智能体是 EvoFlow 的核心交互入口，后续创建智能体、技能、自动化等操作都可以在对话中通过自然语言指令完成。

```text
Python 项目依赖管理确实是个值得深入讨论的话题。让我为你系统梳理一下主流方案及其优缺点。

## 主流方案概览

### 1. **pip + requirements.txt**（传统方案）

**优点：**
- 简单直观，入门门槛低
- 几乎所有 Python 开发者都熟悉
- 工具链成熟，生态支持完善

**缺点：**
- 无法锁定依赖的确切版本（transitive dependencies）
- 不同环境可能产生不同的依赖树
- 手动维护繁琐，容易遗漏

---

### 2. **pipenv**（Pip + Virtualenv 的整合）

**优点：**
- 自动创建和管理虚拟环境
- 生成 `Pipfile` 和 `Pipfile.lock` 锁定依赖
- 安全性检查（`pipenv check`）

**缺点：**
- 性能较慢，大型项目体验差
- 锁定文件有时难以解析
- 社区热度有所下降

---

### 3. **poetry**（当前最推荐）

**优点：**
- 统一的 `pyproject.toml` 配置
- 精准的依赖锁定（`poetry.lock`）
- 内置虚拟环境管理
- 支持打包和发布到 PyPI
- 依赖解析算法更优

**缺点：**
- 学习曲线略陡
- 部分边缘场景（如 C 扩展）配置复杂
- 与某些 CI/CD 工具集成需适配

---

### 4. **conda**（数据科学首选）

**优点：**
- 不仅管理 Python 包，还管理非 Python 依赖（C/C++ 库）
- 预编译二进制包，安装快
- 环境隔离彻底

**缺点：**
- 包仓库相对 PyPI 较小
- 与 pip 混用容易出问题
- 体积较大

---

### 5. **uv**（新兴高性能工具）

**优点：**
- Rust 编写，速度极快（比 pip 快 10-100 倍）
- 兼容 pip/requirements.txt 生态
- 内置虚拟环境管理
- 支持锁定文件

**缺点：**
- 相对较新，生态仍在成熟
- 部分高级功能还在开发中

---

### 6. **pdm**（PEP 582 倡导者）

**优点：**
- 遵循现代 PEP 标准
- 支持 `__pypackages__` 本地安装（无需虚拟环境）
- 类似 poetry 的体验

**缺点：**
- 社区规模相对较小
- `__pypackages__` 模式尚未广泛支持

---

## 选择建议

| 场景 | 推荐工具 |
|------|---------|
| 通用 Web/工具项目 | **Poetry** |
| 数据科学/ML | **Conda** 或 **Poetry** |
| 追求极致性能 | **uv** |
| 简单脚本/学习 | **pip + venv** |
| 遗留项目维护 | **pip-tools** |

---

1. **始终使用锁定文件**（lock file）确保可复现构建

4. **CI/CD 中缓存依赖**：加速构建流程
```

```text
## 🏆 中型团队推荐：**Poetry**

### 推荐理由

| 考量因素 | Poetry 优势 |
|:---|:---|
| **团队协作** | `pyproject.toml` 单文件配置，版本控制友好 |
| **新人 onboarding** | 文档完善，社区案例丰富 |
| **可复现构建** | `poetry.lock` 精确锁定所有依赖树 |
| **发布流程** | 内置 `poetry build` + `poetry publish`，简化 CI/CD |
| **依赖冲突解决** | 智能解析算法，比 pipenv 更可靠 |
| **生态兼容** | 完全兼容 PyPI，与现有工具链无缝集成 |

### 典型工作流

`​``bash
# 初始化项目
poetry new my-project && cd my-project

# 添加依赖
poetry add fastapi sqlalchemy
poetry add --group dev pytest black ruff

# 安装（自动创建虚拟环境）
poetry install

# 锁定并导出（CI/CD 用）
poetry lock && poetry export -f requirements.txt -o requirements.txt
`​``

- 若团队**极度追求构建速度** → **uv**（未来趋势，但生态仍在成熟）
```
